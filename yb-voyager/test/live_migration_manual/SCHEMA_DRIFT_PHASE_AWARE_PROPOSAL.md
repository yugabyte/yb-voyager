# Schema Drift Analysis — Phase-Aware Proposal

This is a proposed presentation format for the schema drift feature analysis. It keeps the important product view first: **what DDL changed, how we detect it, what happens in Forward and Fallback, and what solution Voyager can offer**.

## 1. How To Read This

### 1.1 Phases

| Phase | Export side | Import side | Commands |
|---|---|---|---|
| **Forward** | Source PostgreSQL | Target YugabyteDB | `export data` + `import data` |
| **Fallback** | Target YugabyteDB | Source PostgreSQL | `export data from target` + `import data to source` |

### 1.2 Detection Types

| Detection | Meaning in Forward | Meaning in Fallback |
|---|---|---|
| **S vs S** | Live Source vs Source baseline; detects Source-side/export-side drift | Live Source vs Source baseline; detects import-side drift during fallback |
| **T vs T** | Live Target vs Target baseline; detects Target-side/import-side drift | Live Target vs Target baseline; detects Target-side/export-side drift |
| **S vs T** | Source/Target parity check and error diagnosis | Target/Source parity check and error diagnosis |

### 1.3 Proposed Solution Language

The `Proposed solution` cells are written as product options, not just manual workaround notes.

| Solution style | Meaning |
|---|---|
| **Detect/report only** | Voyager can identify the drift and tell the user what happened, but does not modify migration state. |
| **User-started new migration** | User starts a separate/new migration path. This is full-data support only if it includes snapshot + CDC. |
| **Partial built-in handling** | Voyager could patch internal metadata/config/publication and capture future changes only. Rows missed before the patch need manual backfill. |
| **Full built-in handling** | Voyager updates capture metadata and snapshots/backfills old rows before continuing CDC. This is the ideal level, but only listed when feasible or as a future product target. |
| **DDL + resume** | Apply matching DDL on the lagging side, then resume. Queued events drain, so no row loss if handled promptly. |

### 1.4 Severity Rubric

A single, phase-aware rubric is applied to every row. Reviewers can use this to audit each severity assignment.

| Severity | Definition |
|---|---|
| **P0** | Pipeline-level break: export restart fails, or import cannot recover via DDL+resume. **Or** silent data loss with no Voyager-supported recovery path in this phase. |
| **P1** | Apply error or capture skip that is recoverable via **DDL + resume** on the lagging side, **or** recoverable via a **user-started supplemental migration** without restarting the main migration. |
| **P2** | No row-apply error. Parity-only matters before cutover or cutback. |

Phase-aware consequences of this rubric (called out so the matrix below stays consistent):

- Forward `Create new table with PK`, `Add partition leaf`, `New partitioned parent`, `DEFAULT PARTITION`, multi-leaf add → **P1** because supplemental migration recovers cleanly.
- The same DDLs in **Fallback** → **P0** because there is no supplemental path; metadata surgery captures only future rows.
- **Detach partition** — Doc assumes the user applies **matching detach DDL** on Source and YugabyteDB so partition **topology matches** after the change (**P0**): the stream still encodes **leaf → parent** routing while Target's parent no longer covers the detached range, so apply stalls with partition-routing errors and queued events do not drain until topology is reconciled or the migration is restarted. **Caveat:** if detach **order or completeness** differs between sides, the same class of errors often appears with **messier diagnosis**; recovery is still **reconcile Target (or Source) to the event shape** or **restart/new migration**—not modeled as a separate matrix row.
- `Drop strict column` is split: import side has a default = **P2**; no default = **P1**.
- `Drop PK / change conflict key` is split: drop without replacement = **P0** Forward; change PK columns = **P1** Forward, **P0** Fallback (YB-blocked).
- `Add identity column` carries **P1** for column-missing crash and **P2** for sequence/value parity.

## 2. Severity-First Drift Matrix

### 2.1 P0 — Critical / No Clean Recovery

These cases either break the pipeline (export restart fails or import cannot recover via DDL+resume) or cause silent data loss with no Voyager-supported recovery path **in this phase**. Forward-side DDLs that have a clean supplemental-migration recovery are listed in P1 under their Forward row instead.

| DDL case | Phase | Detection | Behavior / consequence | Proposed solution |
|---|---|---|---|---|
| **Create new table with PK** | Fallback | `T vs T`; `S vs T` for parity | New YB table is outside fallback capture. Rows inserted on YB are not exported. Creating the table on PG only fixes schema parity; it does not replay rows. There is no supplemental-migration path in fallback. | **Partial built-in handling possible:** Voyager could patch metadata/config/name-registry/publication so future YB rows stream. Rows written before patching need manual backfill. **Full built-in handling** would also snapshot/backfill existing YB rows before CDC continues. Otherwise restart/new migration. |
| **Create new table without PK** | Forward | `S vs S`; `S vs T` for parity | Rows from the new Source table are not captured. If the user tries to add it without a PK to live capture, Voyager rejects it with the no-PK guardrail. | **Detect/report only by default:** explain that the table is outside live CDC because it has no PK. **User-started new migration** becomes possible only after the user adds a PK. Partial/full built-in handling should not add it to live CDC while it remains without a PK. |
|  | Fallback | `T vs T`; `S vs T` for parity | Rows from the new YB table are not captured in fallback. Unlike a PK table, this table cannot be added to the YB publication/capture as-is. | **Detect/report only by default:** explain that the new YB table cannot enter fallback CDC without a PK. After adding a PK, treat it like the PK-table case (partial/full built-in handling). |
| **Rename captured table** | Forward | `S vs S`; restart failure diagnosed by stored metadata | The old name remains in Voyager metadata. Rows written under the new name are not handled by the main migration. If `export data` is restarted, it fails because the old table name no longer exists. | **Detect/report as critical:** warn that the main migration metadata still points to the old name. **User-started new migration** or a separate supplemental migration can snapshot+stream the renamed table, but it does not repair the original main migration. **Partial built-in handling** would patch stored table metadata/name registry so future restarts use the new name; rows missed before the patch still need backfill. **Full built-in handling** would do metadata repair plus snapshot/backfill of rows written under the new name before CDC resumes. |
|  | Fallback | `T vs T`; restart failure diagnosed by stored metadata | The old YB table name remains in fallback metadata. Rows written after the rename are not exported. Restart of `export data from target` fails because metadata still references the old name. | **Detect/report as critical:** identify that fallback capture metadata still references the old YB table. **Partial built-in handling** could patch metadata/config/name-registry/publication so only future renamed-table events stream; rows inserted before patching still need backfill. **Full built-in handling** would backfill those rows from the renamed YB table, then continue CDC. Otherwise restart/new migration. |
| **Drop captured table** | Forward | `S vs S` | The captured Source table disappears but Voyager metadata still includes it. The running exporter may continue for other tables, but a later `export data` restart fails when Voyager looks up the dropped table. | **Detect + scope removal (built-in):** Voyager detects that the table no longer exists on the source but is still listed in migration scope (**critical**), confirms that it should **leave** the migration, then **removes it from stored capture metadata** (`meta.db`, Debezium `table.include.list` / connector scope, publication, `name_registry.json`, and related on-disk artifacts as needed) so **`export data` can restart.** |
|  | Fallback | `T vs T` | YB blocks the drop while the table is in the publication. If removed from publication and dropped, the fallback exporter crashes; restart also fails because fallback metadata still contains the dropped table. | **Detect + scope removal (built-in):** Voyager detects that the YugabyteDB table is gone but still listed in fallback scope (**critical**), confirms removal from the migration, then **drops it from fallback capture metadata** (stored config, publication, `name_registry`, related artifacts for **`export data from target`**) so **restart can succeed.** |
| **Add partition leaf** | Fallback | `T vs T`; restart warning | The new YugabyteDB leaf is outside the publication/capture set. Existing captured leaves continue to stream, but rows routed to the new leaf are silently skipped. Restart warns about the new leaf but does not add it to capture. Creating the matching leaf on PostgreSQL only fixes schema parity. | **Same options as Create new table with PK (Fallback):** partial built-in handling; full built-in handling; otherwise restart/new migration. |
| **New partitioned parent (root + leaves)** | Fallback | `T vs T`; `S vs T` for parity | The new YB parent and all of its initial leaves are outside fallback capture. Same silent skip as a fully new table, but covers the whole subtree. | Same three paths as Create new table with PK (Fallback): **1)** Supplemental fallback migration — dedicated target→source migration in a fresh `export-dir` (snapshot + changes enabled in **export data from target**), table list scoped to the **new subtree** (**root or all leaves** — verified either way), snapshot + ongoing CDC for that scope. **2)** Partial built-in handling — patch fallback scope for **root + each leaf** (meta, connector, publication, `name_registry`), restart **export data from target**; backfill pre-scope rows on YugabyteDB across the subtree (and gap if DML was not paused). **3)** Full built-in handling — add the **subtree** in the same fallback migration, snapshot/load the **entire subtree's** existing data into PostgreSQL, then continue streaming. |
| **DEFAULT PARTITION mid-run** | Fallback | `T vs T`; restart warning | The new YB default leaf catches rows that no longer match an explicit range. Without capture inclusion, those rows are silently skipped. | **Partial built-in handling possible:** add the default leaf to fallback metadata/publication so future routed rows stream. **Full built-in handling** also backfills rows already routed to the default leaf. |
| **Drop primary key without replacement (M)** | Forward | `S vs S`; `S vs T` for diagnosis | The CDC event key collapses. Voyager builds invalid `INSERT … ON CONFLICT ()` (`SQLSTATE 42601`). On restart, the no-PK guardrail blocks export with `Currently voyager does not support live-migration for tables without a primary key`. | **Detect/report as critical before data loss.** Recovery is to **re-add a PK on Source** (preferred) or use **restart/new migration** excluding the table. Removing the table from capture mid-run is unsupported and risky (queue + schema-registry stale state). |
| **Change PK columns (O)** | Fallback | `T vs T` | YB rejects `ALTER TABLE … DROP CONSTRAINT … PRIMARY KEY` for tables in CDC: `Cannot rewrite a table that is a part of CDC`. Voyager's key-mismatch path is not reached. | **Detect/report only:** blocked by the database before Voyager can support a CDC recovery path. Partial/full built-in handling is not applicable until YB allows safe CDC-table key rewrite. |
| **Detach partition** | Forward | `S vs S`; `S vs T` | **Scope:** matching detach on Source and YugabyteDB (same topology outcome on both sides). Source still emits **leaf** events (the leaf is still in the publication and `table.include.list`). Voyager's rename map rewrites them to the partitioned parent on Target, but that parent **no longer has a partition** for the detached range, so apply fails with `no partition of relation "part_t" found for row`. **Resume does not help** until the failure shape changes, because queued events still route to a parent that lost the slot. **Caveat:** if detach lands **only on one side** or **out of order**, behaviour can include a **silent divergence** window first; once topology no longer matches the rewrite, you still hit the same partition-routing stall. | **Detect/report partition topology drift.** If Target can be **reconciled to the queued event shape**, **DDL + resume:** **re-attach** the leaf on Target with the **matching range** (see manual scenario **T**). Otherwise **restart/new migration** with topology aligned to capture and rewrite rules. |

### 2.2 P1 — Recoverable Or Partially Supported

These cases either fail at apply with a clear error and are recoverable via **DDL + resume**, or they silently skip rows but can be fully recovered via a user-started supplemental migration without restarting the main migration.

| DDL case | Phase | Detection | Behavior / consequence | Proposed solution |
|---|---|---|---|---|
| **Create new table with PK** | Forward | `S vs S`; `S vs T` for parity | New Source table is outside the main capture set. Rows written to it are silently skipped by the main migration; the main exporter does not crash. | **User-started supplemental migration:** start a separate `yb-voyager` migration in a fresh `export-dir` for the new table; it snapshots existing rows and streams future changes. **Partial built-in handling** could add the table to capture for future rows only. **Full built-in handling** would snapshot+stream inside the main migration. If the table is intentionally out of scope, **detect/report only**. |
| **Add partition leaf** | Forward | `S vs S`; restart warning | The new Source leaf is outside the publication/capture set. Rows routed to that leaf are silently skipped. Restart warns about the new leaf but does not add it to capture. | **Same options as Create new table with PK (Forward):** user-started supplemental migration; partial built-in handling; full built-in handling. If the leaf is intentionally out of scope, **detect/report only**. |
| **New partitioned parent (root + leaves)** | Forward | `S vs S`; restart warning | A new partitioned parent and its initial leaves are outside the main capture. Rows on any leaf are silently skipped. | Same three paths as Create new table with PK (Forward): **1)** Supplemental migration — separate migration in a fresh `export-dir` with the **new partitioned parent subtree** in the table list (**root only** or **all leaves** — verified either way). **2)** Partial built-in handling — Voyager extends capture scope to the **parent and each leaf** and restarts export; user backfills rows missed before the subtree was in scope (and any gap if DML was not paused). **3)** Full built-in handling — Voyager adds the **subtree** in the same migration, loads existing data for the **whole partitioned parent** to the target, then continues streaming. If the subtree is intentionally out of scope, **detect/report only**. |
| **DEFAULT PARTITION mid-run** | Forward | `S vs S`; restart warning | Rows that don't match any explicit range are routed to the default leaf, which is outside capture. | **User-started supplemental migration** for the default leaf. **Partial built-in handling** can add the default leaf to capture for future rows; **full built-in handling** also backfills already-routed rows. |
| **Add column** (incl. partitioned parent — AD) | Forward | `S vs S`; `S vs T` | Source emits new column; Target lacks it; import panics on unknown column. For a partitioned parent, leaf events all carry the new column and a single ALTER on the Target parent is sufficient (PG/YB propagate to leaves). | **DDL + resume:** detect the missing Target column and guide/apply `ALTER TABLE … ADD COLUMN`. For partitioned parents, one ALTER on Target's parent is enough. Queued events drain after alignment. |
|  | Fallback | `T vs T`; `S vs T` | YB emits new column; PG lacks it; `import data to source` panics on unknown column. | **DDL + resume:** guide/apply compatible PG `ADD COLUMN` and resume. |
| **Add column on Target only (B)** | Forward | `T vs T`; `S vs T` | Target-only DDL is harmless on its own. Once Source also gains the column, the **first** import resume can fail because the on-disk Debezium schema JSON snapshot is stale (`Column … not found in table schema`). A second restart of `import data` reloads the schema files and succeeds. | **Detect/report and DDL + resume:** advise the user that target-only column is harmless until Source also adds the column. After Source ALTER, Voyager should guide a **single extra restart of `import data`** so the schema registry reloads from disk. **Full built-in handling** would invalidate the schema-registry cache when fresh schemas are written. |
| **Rename column** | Forward | `S vs S`; `S vs T` | Source event uses the new column name; Target still has the old name; import panics. | **DDL + resume:** detect the rename pair and guide/apply Target `RENAME COLUMN`, then resume. |
|  | Fallback | `T vs T`; `S vs T` | YB event uses the new name; PG still has the old name; fallback import panics. | **DDL + resume:** guide/apply PG `RENAME COLUMN` to match YB, then resume. |
| **Incompatible type change** | Forward | `S vs S`; `S vs T` | Source value no longer fits Target type; import fails with type conversion error. | **DDL + resume when safe:** guide compatible Target `ALTER COLUMN TYPE`, using `USING` if needed. Auto-apply should be conservative because lossy/manual conversions need user approval. |
|  | Fallback | `T vs T`; `S vs T` | Same pattern with YB as export side and PG as import side. | **DDL + resume when safe:** guide compatible PG type alignment. |
| **Drop strict column without default on import side (L)** | Forward | `S vs T`; `S vs S` for dropped column | Source omits the column; Target still has `NOT NULL` without a default; import fails with `23502`. | **DDL + resume:** drop the Target column, drop `NOT NULL`, or add a default. Once aligned, queued events drain. |
|  | Fallback | `S vs T`; `T vs T` for dropped column | YB omits the column; if PG still has `NOT NULL` without default, fallback import fails with `23502`. | **DDL + resume:** drop/relax/default on PG. |
| **Add NOT NULL on import side (R)** | Forward | `T vs T`; `S vs T` | Target column is `NOT NULL` while Source column is nullable. NULL inserts from Source fail with `23502`. | **DDL + resume:** drop/relax `NOT NULL` on Target, or enforce the same constraint on Source and repair NULL data, then resume. |
| **Add CHECK on import side** | Forward | `T vs T`; `S vs T` | Target CHECK can reject Source-valid rows; import fails with `23514`. `session_replication_role = replica` does not disable CHECK. | **Detect/report plus DDL + resume:** relax/drop the Target CHECK or enforce the same CHECK on Source and repair invalid data before resume. Auto-apply should require user approval because validation semantics change. |
|  | Fallback | `S vs S`; `S vs T` | PG CHECK can reject YB-valid rows. | **Detect/report plus DDL + resume:** relax/drop PG CHECK or enforce the same CHECK on YB and repair invalid data. |
| **Add identity column (column-missing crash)** | Forward | `S vs S`; `S vs T` | Source identity column appears in CDC; Target missing column causes import panic. (Sequence/value parity is a separate parity-only concern — see P2.) | **DDL + resume:** add a compatible plain column on Target and resume. **Full built-in handling** also reconciles identity/sequence state; otherwise the parity sub-row in P2 captures the remaining work. |
|  | Fallback | `T vs T`; `S vs T` | Same shape with YB as export side and PG as import side. | **DDL + resume:** add a compatible PG column and resume. |
| **Add enum value** | Forward | `S vs S`; `S vs T` | Source emits new enum label; Target enum lacks it; import fails. | **DDL + resume:** guide/apply `ALTER TYPE … ADD VALUE` on Target before resume. Good candidate for built-in handling. |
|  | Fallback | `T vs T`; `S vs T` | Expected same pattern with YB enum value missing on PG. | **DDL + resume:** guide/apply `ALTER TYPE … ADD VALUE` on PG. |
| **Drop unique on export side while import side keeps it** | Forward | `S vs T`; `S vs S` for dropped unique | Source allows duplicates; Target still has unique; import fails with `23505`. | **DDL + resume or data repair:** drop/align Target uniqueness if duplicates are intended, or repair duplicate data before resume. |
|  | Fallback | `S vs T`; `T vs T` for dropped unique | YB allows duplicates; PG still has unique; fallback import fails with `23505`. | **DDL + resume or data repair:** same product choice with PG as import side. |
| **Relax NOT NULL on export side** | Forward | `S vs T` | Source accepts/sends NULL; Target remains strict; import fails with `23502`. | **DDL + resume:** guide/apply Target `DROP NOT NULL` or default strategy. |
|  | Fallback | `S vs T` | YB accepts/sends NULL; PG remains strict; fallback import fails with `23502`. | **DDL + resume:** guide/apply PG `DROP NOT NULL` or default strategy. |
| **Change PK columns (O)** | Forward | `S vs S`; `S vs T` | The CDC event key shape no longer matches Target's PK/unique. Import can fail because the generated `ON CONFLICT (col)` no longer matches a Target constraint. | **DDL + resume:** align Target PK/unique to the new shape before queued events apply. **Restart/new migration** if the new key cannot be made compatible mid-flight. |

### 2.3 P2 — Advisory / Parity

These cases do not crash the pipeline. They still matter because they change application behavior after cutover or cutback.

| DDL case | Phase | Detection | Behavior / consequence | Proposed solution |
|---|---|---|---|---|
| **Generated column added on export side** | Forward | `S vs S`; `S vs T` | PG logical replication does not publish generated columns by default. Segment files omit the column; no import failure. Plain Target column remains unset. | **Detect/report for parity:** explain the column is not in the CDC payload. Guide creation of a matching generated column on Target. If the destination must store materialized values in a plain column, manual backfill is required. |
|  | Fallback | `T vs T`; `S vs T` | Expected similar depending on YB logical replication behavior; treat as parity-sensitive. | **Detect/report for parity:** prefer matching generated definition on PG where supported; manual compatibility/backfill plan otherwise. |
| **Add column with inline CHECK on Source (AC)** | Forward | `S vs S`; `S vs T` | The CHECK is evaluated only on Source; CDC payload does not carry CHECK constraints. Apply on Target proceeds normally. | **Detect/report for parity:** guide/apply matching CHECK on Target before cutover if validation parity is required. |
| **Drop strict column with default on import side (K)** | Forward | `S vs T`; `S vs S` for dropped column | Source omits the column; Target still has `NOT NULL` but with a `DEFAULT`. Apply succeeds because the engine fills the default for the missing column. | **Detect/report for parity:** guide Target column drop before cutover, or accept that Target retains the column with default-filled rows. |
| **Add identity column — sequence parity** | Both | `S vs S`, `T vs T`, `S vs T` | Even after the column is aligned, Target sequence state may not match Source sequence advances when CDC supplies explicit values. After cutover, `nextval` on Target can collide with values already produced on Source. | **Detect/report for parity:** verify and reset Target sequence before cutover. **Full built-in handling** would reconcile identity/sequence state automatically. |
| **Add unique on export side** | Forward | `S vs S`; `S vs T` for parity | Source enforces uniqueness before CDC. Target missing unique does not break apply for valid rows. | **Detect/report for parity:** guide/apply matching unique on Target before cutover. **Full built-in handling** is feasible if Target data is already unique. |
|  | Fallback | `T vs T`; `S vs T` for parity | YB enforces uniqueness; PG missing unique does not break apply for valid rows. | **Detect/report for parity:** guide/apply matching unique on PG before cutback. |
| **Add FK / FK change** | Forward | `S vs S` or `T vs T`; `S vs T` for parity | FK enforcement is disabled during CDC apply via `session_replication_role = replica`; no apply failure for source-valid rows. | **Detect/report for parity:** guide/apply matching FK on Target before normal application writes resume. **Full built-in handling** must validate existing data before adding the FK. |
|  | Fallback | `T vs T` or `S vs S`; `S vs T` for parity | Same: FK triggers disabled during apply, but matter after cutback. | **Detect/report for parity:** guide/apply matching FK on PG before cutback. |
| **Drop nullable column** | Forward | `S vs S`; `S vs T` | Source omits column; nullable Target column fills NULL/default; no crash. | **Detect/report for parity:** guide Target column drop before cutover, or mark the Target-only column as intentionally ignored. |
|  | Fallback | `T vs T`; `S vs T` | YB omits column; nullable PG column fills NULL/default; no crash. | **Detect/report for parity:** guide PG column drop before cutback, or mark intentionally ignored. |
| **ATTACH PARTITION of standalone onto captured parent (W)** | Forward | `S vs S`; `S vs T` | Symmetric DDL: when the leaf already exists as a standalone table on both sides and is attached as a partition on both sides, events stream to the parent and apply normally — no failure was observed. | **Detect/report for parity:** confirm the leaf is attached at the same range on both sides before treating it as captured. |
| **Default value change** | Forward | `S vs S`; `S vs T` | CDC carries explicit values; no import failure. Future app inserts after cutover can diverge. | **Detect/report for parity:** guide/apply matching Target default before cutover. |
|  | Fallback | `T vs T`; `S vs T` | Same with YB as active side. | **Detect/report for parity:** guide/apply matching PG default before cutback. |
| **Switch REPLICA IDENTITY (BC)** | Forward | `S vs S`; `S vs T` | Changes the CDC event key shape and therefore Voyager's `ON CONFLICT` target. Not separately verified for recovery; YB CDC requires PK so this matters most when source-side replica identity moves to a non-PK unique index. | **Detect/report only:** flag when REPLICA IDENTITY changes mid-run. Until verified, no automated DDL + resume is proposed. |
| **Sequence property change** | Both | `S vs S`, `T vs T`, `S vs T` | Voyager may sync sequence values, but not all sequence DDL properties. | **Detect/report for parity:** distinguish sequence value sync from sequence definition drift. Verify next values and align sequence options before cutover/cutback. |
| **Views / functions / policies / non-row-shape objects** | Both | `S vs S`, `T vs T`, `S vs T` | CDC does not carry these object definitions. Runtime behavior can diverge later. | **Detect/report only or guided DDL apply:** outside row CDC. No CDC recovery/backfill concept applies. |
| **Index-only changes (non-unique)** | Both | `S vs S`, `T vs T`, `S vs T` as needed | Usually no row correctness issue; may affect performance. Unique indexes are handled separately. | **Detect/report for advisory impact:** classify as performance/operational drift. Guided DDL apply is optional. |

The running exporter may continue for other tables, but a later `export data` restart fails when Voyager looks up the dropped table. | **Detect + scope removal:** confirm table left migration, strip from stored capture metadata so **`export data` restart** works. |
|  | P0 | Fallback | `T vs T` | YB blocks the drop while the table is in the publication. If removed from publication and dropped, the fallback exporter crashes; restart also fails because fallback metadata still contains the dropped table. | **Detect + scope removal:** confirm, strip from fallback metadata/publication/registry so **`export data from target` restart** works. |
| **Add partition leaf** | P1 | Forward | `S vs S`; restart warning | The new Source leaf is outside the publication/capture set. Rows routed to that leaf are silently skipped. Restart warns about the new leaf but does not add it to capture. | **Same options as Create new table with PK (Forward):** user-started supplemental migration; partial built-in handling; full built-in handling. If intentionally out of scope, **detect/report only**. |
|  | P0 | Fallback | `T vs T`; restart warning | The new YugabyteDB leaf is outside the publication/capture set. Existing captured leaves continue to stream, but rows routed to the new leaf are silently skipped. Restart warns about the new leaf but does not add it to capture. Creating the matching leaf on PostgreSQL only fixes schema parity. | **Same options as Create new table with PK (Fallback):** partial built-in handling; full built-in handling; otherwise restart/new migration. |
| **New partitioned parent (root + leaves)** | P1 | Forward | `S vs S`; restart warning | A new partitioned parent and its initial leaves are outside the main capture. Rows on any leaf are silently skipped. | **User-started supplemental migration:** verified to work whether the table list is the root or the leaves. **Partial/full built-in handling** mirrors the new-table case applied to the whole subtree. |
|  | P0 | Fallback | `T vs T`; `S vs T` for parity | The new YB parent and all of its initial leaves are outside fallback capture. Same silent skip as a fully new table, but covers the whole subtree. | **Partial built-in handling possible:** patch metadata for root + each leaf to capture future events. **Full built-in handling** would snapshot/backfill the entire subtree. |
| **DEFAULT PARTITION mid-run** | P1 | Forward | `S vs S`; restart warning | Rows that don't match any explicit range are routed to the default leaf, which is outside capture. | **User-started supplemental migration** for the default leaf. **Partial built-in handling** can add the default leaf to capture for future rows; **full built-in handling** also backfills already-routed rows. |
|  | P0 | Fallback | `T vs T`; restart warning | The new YB default leaf catches rows that no longer match an explicit range. Without capture inclusion, those rows are silently skipped. | **Partial built-in handling possible:** add the default leaf to fallback metadata/publication so future routed rows stream. **Full built-in handling** also backfills rows already routed to the default leaf. |
| **Detach partition** | P0 | Forward | `S vs S`; `S vs T` | After symmetric detach on both sides, Source still emits leaf events. Voyager rewrites them to the parent on Target, but Target's parent no longer covers the detached range, so apply fails with `no partition of relation "part_t" found for row`. Queued events cannot drain. | **Detect/report partition topology drift:** re-attach the leaf on Target with the matching range so apply can resume, or **restart/new migration** with the new topology. |
| **ATTACH PARTITION of standalone onto captured parent (W)** | P2 | Forward | `S vs S`; `S vs T` | Symmetric DDL: leaf attached at the same range on both sides → events route to the parent on Target and apply normally. | **Detect/report for parity:** confirm symmetric attach at the same range before treating the leaf as captured. |

### 3.2 Column Shape Changes

| DDL case | Severity | Phase | Detection | Behavior / consequence | Proposed solution |
|---|---|---|---|---|---|
| **Add column** (incl. partitioned parent — AD) | P1 | Forward | `S vs S`; `S vs T` | Source emits new column; Target lacks it; import panics on unknown column. For a partitioned parent, leaf events all carry the new column and a single ALTER on the Target parent is sufficient (PG/YB propagate to leaves). | **DDL + resume:** detect the missing Target column and guide/apply `ALTER TABLE … ADD COLUMN`. For partitioned parents, one ALTER on Target's parent is enough. Queued events drain after alignment. |
|  | P1 | Fallback | `T vs T`; `S vs T` | YB emits new column; PG lacks it; `import data to source` panics on unknown column. | **DDL + resume:** guide/apply compatible PG `ADD COLUMN` and resume. |
| **Add column on Target only (B)** | P1 | Forward | `T vs T`; `S vs T` | Target-only DDL is harmless on its own. Once Source also gains the column, the **first** import resume can fail because the on-disk Debezium schema JSON snapshot is stale (`Column … not found in table schema`). A second restart of `import data` reloads the schema files and succeeds. | **Detect/report and DDL + resume:** advise the user that target-only column is harmless until Source also adds the column. After Source ALTER, Voyager should guide a **single extra restart of `import data`** so the schema registry reloads from disk. **Full built-in handling** would invalidate the schema-registry cache when fresh schemas are written. |
| **Rename column** | P1 | Forward | `S vs S`; `S vs T` | Source event uses the new column name; Target still has the old name; import panics. | **DDL + resume:** detect the rename pair and guide/apply Target `RENAME COLUMN`, then resume. |
|  | P1 | Fallback | `T vs T`; `S vs T` | YB event uses the new name; PG still has the old name; fallback import panics. | **DDL + resume:** guide/apply PG `RENAME COLUMN` to match YB, then resume. |
| **Incompatible type change** | P1 | Forward | `S vs S`; `S vs T` | Source value no longer fits Target type; import fails with type conversion error. | **DDL + resume when safe:** guide compatible Target `ALTER COLUMN TYPE`, using `USING` if needed. Auto-apply should be conservative because lossy/manual conversions need user approval. |
|  | P1 | Fallback | `T vs T`; `S vs T` | Same pattern with YB as export side and PG as import side. | **DDL + resume when safe:** guide compatible PG type alignment. |
| **Drop strict column with default on import side (K)** | P2 | Forward | `S vs T`; `S vs S` for dropped column | Source omits the column; Target still has `NOT NULL` but with a `DEFAULT`. Apply succeeds because the engine fills the default for the missing column. | **Detect/report for parity:** guide Target column drop before cutover, or accept that Target retains the column with default-filled rows. |
| **Drop strict column without default on import side (L)** | P1 | Forward | `S vs T`; `S vs S` for dropped column | Source omits the column; Target still has `NOT NULL` without a default; import fails with `23502`. | **DDL + resume:** drop the Target column, drop `NOT NULL`, or add a default. Once aligned, queued events drain. |
|  | P1 | Fallback | `S vs T`; `T vs T` for dropped column | YB omits the column; if PG still has `NOT NULL` without default, fallback import fails with `23502`. | **DDL + resume:** drop/relax/default on PG. |
| **Drop nullable column** | P2 | Forward | `S vs S`; `S vs T` | Source omits column; nullable Target column fills NULL/default; no crash. | **Detect/report for parity:** guide Target column drop before cutover, or mark the Target-only column as intentionally ignored. |
|  | P2 | Fallback | `T vs T`; `S vs T` | YB omits column; nullable PG column fills NULL/default; no crash. | **Detect/report for parity:** guide PG column drop before cutback, or mark intentionally ignored. |
| **Generated column added on export side** | P2 | Forward | `S vs S`; `S vs T` | PG logical replication does not publish generated columns by default. Segment files omit the column; no import failure. Plain Target column remains unset. | **Detect/report for parity:** explain the column is not in the CDC payload. Guide creation of a matching generated column on Target. If the destination must store materialized values in a plain column, manual backfill is required. |
|  | P2 | Fallback | `T vs T`; `S vs T` | Expected similar depending on YB logical replication behavior; treat as parity-sensitive. | **Detect/report for parity:** prefer matching generated definition on PG where supported; manual compatibility/backfill plan otherwise. |
| **Add column with inline CHECK on Source (AC)** | P2 | Forward | `S vs S`; `S vs T` | The CHECK is evaluated only on Source; CDC payload does not carry CHECK constraints. Apply on Target proceeds normally. | **Detect/report for parity:** guide/apply matching CHECK on Target before cutover if validation parity is required. |
| **Add identity column (column-missing crash)** | P1 | Forward | `S vs S`; `S vs T` | Source identity column appears in CDC; Target missing column causes import panic. (Sequence/value parity is a separate parity-only concern — see P2.) | **DDL + resume:** add a compatible plain column on Target and resume. **Full built-in handling** also reconciles identity/sequence state; otherwise the parity sub-row in P2 captures the remaining work. |
|  | P1 | Fallback | `T vs T`; `S vs T` | Same shape with YB as export side and PG as import side. | **DDL + resume:** add a compatible PG column and resume. |
| **Add identity column — sequence parity** | P2 | Both | `S vs S`, `T vs T`, `S vs T` | Even after the column is aligned, Target sequence state may not match Source sequence advances when CDC supplies explicit values. After cutover, `nextval` on Target can collide with values already produced on Source. | **Detect/report for parity:** verify and reset Target sequence before cutover. **Full built-in handling** would reconcile identity/sequence state automatically. |
| **Add enum value** | P1 | Forward | `S vs S`; `S vs T` | Source emits new enum label; Target enum lacks it; import fails. | **DDL + resume:** guide/apply `ALTER TYPE … ADD VALUE` on Target before resume. Good candidate for built-in handling. |
|  | P1 | Fallback | `T vs T`; `S vs T` | Expected same pattern with YB enum value missing on PG. | **DDL + resume:** guide/apply `ALTER TYPE … ADD VALUE` on PG. |
| **Relax NOT NULL on export side** | P1 | Forward | `S vs T` | Source accepts/sends NULL; Target remains strict; import fails with `23502`. | **DDL + resume:** guide/apply Target `DROP NOT NULL` or default strategy. |
|  | P1 | Fallback | `S vs T` | YB accepts/sends NULL; PG remains strict; fallback import fails with `23502`. | **DDL + resume:** guide/apply PG `DROP NOT NULL` or default strategy. |
| **Default value change** | P2 | Forward | `S vs S`; `S vs T` | CDC carries explicit values; no import failure. Future app inserts after cutover can diverge. | **Detect/report for parity:** guide/apply matching Target default before cutover. |
|  | P2 | Fallback | `T vs T`; `S vs T` | Same with YB as active side. | **Detect/report for parity:** guide/apply matching PG default before cutback. |

## 4. Detection Strategy

| Migration phase | Export-side drift detection | Import-side drift detection | Cross-side diagnosis |
|---|---|---|---|
| Forward | `S vs S` | `T vs T` | `S vs T` |
| Fallback | `T vs T` | `S vs S` | `S vs T` |

Key rules:

- Export-side table/capture inventory drift is the highest-risk class because rows may be skipped before import ever sees them.
- Import-side mismatches are usually recoverable because events are already in the queue and can drain after DDL alignment.
- `S vs T` is best for actionable diagnosis, but it does not always prove capture membership. A table can exist on one side and still not be in Voyager's capture set.

## Open Product Decisions

1. **Phase-aware severity for new objects.** The Forward / Fallback split for `Create new table with PK`, `Add partition leaf`, `New partitioned parent`, and `DEFAULT PARTITION` is asymmetric (P1 Forward, P0 Fallback) because supplemental migration only exists in Forward. Should we invest in a fallback supplemental-migration equivalent so these cases land at P1 in both phases, or accept the asymmetry and ship partial built-in handling for fallback only?
2. **Built-in metadata surgery as a supported feature.** Several P0/P1 cases (Create new table in Fallback, Add partition leaf in Fallback, Rename table in either phase) propose **partial built-in handling** that mirrors today's manual metadata/config/name-registry/publication patching. Should Voyager expose this as a supported product flow with explicit user confirmation, or keep it strictly internal/unsupported and require restart/new migration in customer-facing scenarios?
3. **Auto-apply vs guided DDL.** For all P1 `DDL + resume` rows (Add column, Rename column, Drop strict column without default, Add enum value, Add NOT NULL on import side, etc.), should Voyager auto-apply the proposed DDL with user confirmation, or only print the exact DDL and let the user run it manually? Auto-apply needs to handle Source vs Target identity, schema-registry cache invalidation (Scenario B), and back-out paths.
4. **Explicit "ignored object" outcome.** Several P2 rows (Drop nullable column, Generated column, ATTACH PARTITION of standalone) imply the user may intentionally want the drift to remain. Should Voyager surface this as a first-class supported outcome (mark object as intentionally ignored) so the drift report stops flagging it, instead of relying on detect/report alone?
5. **Identity sequence parity built-in handling.** The P2 sub-row for identity sequence parity describes a real risk that is currently advisory only. Should Voyager add an explicit pre-cutover sequence-reconciliation step for identity columns?
6. **REPLICA IDENTITY drift (BC).** This case is detect-only today because it was not separately verified. Should we run dedicated tests so we can promote this to a concrete handling row, or is detect-only acceptable indefinitely?
